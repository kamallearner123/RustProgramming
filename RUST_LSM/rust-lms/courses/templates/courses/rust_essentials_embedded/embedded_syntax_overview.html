Overview of Rust Syntax for Embedded Systems

Embedded Rust leverages specific language features and syntax that enable safe, efficient low-level programming. This tutorial covers Rust constructs most relevant to embedded development.

1. Essential Attributes for Embedded

#![no_std] - No Standard Library

The most fundamental attribute for embedded Rust - disables the standard library:

#![no_std] #![no_main] use panic_halt as _; #[entry] fn main() -> ! { // Embedded application code loop {} }
Why no_std? The standard library assumes:
  • Operating system (file I/O, threads, networking)
  • Dynamic memory allocation
  • Large binary size footprint
Embedded systems typically lack these, so we use only the core library.

#[entry] - Program Entry Point

use cortex_m_rt::entry; #[entry] fn main() -> ! { // This is the entry point // Note: Returns ! (never type) loop {} }

Key points:

  • Replaces the standard fn main()
  • Must return ! (never returns)
  • Provided by runtime crate (e.g., cortex-m-rt)

#[interrupt] - Interrupt Handlers

use stm32f3xx_hal::interrupt; #[interrupt] fn TIM2() { // Timer 2 interrupt handler // Automatically called when interrupt fires }

#[exception] - Exception Handlers

use cortex_m_rt::exception; #[exception] fn HardFault(ef: &cortex_m_rt::ExceptionFrame) -> ! { panic!("HardFault: {:#?}", ef); }

#[panic_handler] - Custom Panic Behavior

use core::panic::PanicInfo; #[panic_handler] fn panic(info: &PanicInfo) -> ! { // Custom panic behavior // Blink LED rapidly, send to UART, etc. loop {} }

2. The Never Type (!)

The never type ! indicates a function never returns - crucial for embedded main loops:

fn main() -> ! { // Initialize hardware loop { // Main loop never exits } } // Functions that diverge fn halt() -> ! { loop {} } fn reboot() -> ! { unsafe { // Trigger system reset cortex_m::peripheral::SCB::sys_reset(); } }
Why is this important?
  • Embedded systems rarely "exit" - they run forever
  • Compiler knows these functions never return
  • Enables certain optimizations and flow analysis

3. Inline Assembly

For ultra-low-level operations, Rust supports inline assembly:

Basic Inline Assembly

use core::arch::asm; // No operation (NOP) instruction unsafe { asm!("nop"); } // Wait For Interrupt unsafe { asm!("wfi"); } // Data Synchronization Barrier unsafe { asm!("dsb"); }

Assembly with Inputs/Outputs

// Read from register let value: u32; unsafe { asm!( "mrs {}, PRIMASK", out(reg) value ); } // Write to register let new_value: u32 = 1; unsafe { asm!( "msr PRIMASK, {}", in(reg) new_value ); }

Clobbered Registers

unsafe { asm!( "add {0}, {1}, {2}", inout(reg) a, in(reg) b, in(reg) c, options(nomem, nostack) ); }
Use inline assembly sparingly! It's:
  • Target-specific (not portable)
  • Requires unsafe
  • Bypasses compiler optimizations
Most operations have safe abstractions in HAL crates.

4. Volatile Memory Access

Memory-mapped I/O requires volatile operations to prevent compiler optimizations:

Raw Volatile Operations

use core::ptr; // Read from memory-mapped register let value = unsafe { ptr::read_volatile(0x4000_0000 as *const u32) }; // Write to memory-mapped register unsafe { ptr::write_volatile(0x4000_0000 as *mut u32, value); }

Using vcell for Safe Volatile

use vcell::VolatileCell; struct GpioRegisters { moder: VolatileCell, otyper: VolatileCell, ospeedr: VolatileCell, } impl GpioRegisters { fn set_mode(&self, pin: u8, mode: u8) { let mut val = self.moder.get(); val &= !(0b11 << (pin * 2)); val |= (mode as u32) << (pin * 2); self.moder.set(val); } }
Why volatile?
  • Prevents compiler from caching register reads
  • Ensures writes actually happen (not optimized away)
  • Critical for hardware interaction

5. Unsafe Code in Embedded

Embedded development often requires unsafe - here's when and why:

Common Unsafe Operations

// 1. Dereferencing raw pointers (memory-mapped I/O) let gpio_base = 0x4002_0000 as *mut u32; unsafe { *gpio_base = 0x0000_5555; } // 2. Calling unsafe functions unsafe { cortex_m::interrupt::disable(); } // 3. Accessing mutable static variables static mut COUNTER: u32 = 0; unsafe { COUNTER += 1; } // 4. Implementing unsafe traits unsafe impl Send for MyPeripheral {} unsafe impl Sync for MyPeripheral {}

Safe Abstractions Over Unsafe

// Unsafe implementation fn read_register_unsafe(addr: u32) -> u32 { unsafe { ptr::read_volatile(addr as *const u32) } } // Safe wrapper pub struct Register { addr: u32, } impl Register { pub fn read(&self) -> u32 { // Safety: addr is validated at construction unsafe { ptr::read_volatile(self.addr as *const u32) } } pub fn write(&self, value: u32) { // Safety: addr is validated at construction unsafe { ptr::write_volatile(self.addr as *mut u32, value) } } }
Unsafe Guidelines:
  • Keep unsafe blocks small and focused
  • Document safety invariants with // Safety: comments
  • Create safe abstractions over unsafe primitives
  • Use HAL crates when available (they handle unsafe for you)

6. Repr Attributes for Memory Layout

Control struct memory layout for hardware compatibility:

#[repr(C)] - C-Compatible Layout

#[repr(C)] struct GpioRegisters { moder: u32, // Offset: 0x00 otyper: u32, // Offset: 0x04 ospeedr: u32, // Offset: 0x08 pupdr: u32, // Offset: 0x0C } // Now matches hardware register layout

#[repr(packed)] - No Padding

#[repr(C, packed)] struct PacketHeader { version: u8, // No padding between fields flags: u8, length: u16, }
Warning: #[repr(packed)] can cause alignment issues. Accessing fields may require ptr::read_unaligned.

#[repr(transparent)] - Newtype Pattern

#[repr(transparent)] struct Hertz(pub u32); // Same memory layout as u32 // But provides type safety

7. Const Functions and Const Generics

Compile-time computation is valuable in embedded systems:

Const Functions

const fn calculate_prescaler(input_freq: u32, target_freq: u32) -> u32 { input_freq / target_freq - 1 } // Computed at compile time const TIMER_PRESCALER: u32 = calculate_prescaler(72_000_000, 1_000_000);

Const Generics

struct Buffer { data: [u8; N], } impl Buffer { const fn new() -> Self { Self { data: [0; N] } } } // Different sizes, no dynamic allocation let small_buf: Buffer<64> = Buffer::new(); let large_buf: Buffer<1024> = Buffer::new();

8. Type State Pattern

Use types to enforce state at compile time:

// State types struct Disabled; struct Enabled; struct Peripheral { _state: PhantomData, } impl Peripheral { fn enable(self) -> Peripheral { // Enable peripheral Peripheral { _state: PhantomData } } } impl Peripheral { fn read_data(&self) -> u32 { // Can only read when enabled 0 } fn disable(self) -> Peripheral { // Disable peripheral Peripheral { _state: PhantomData } } } // Usage let periph = Peripheral:: { _state: PhantomData }; let periph = periph.enable(); // Now enabled let data = periph.read_data(); // OK // periph.enable(); // Compile error! Already enabled

9. Zero-Sized Types (ZSTs)

Types with no runtime cost - perfect for embedded:

use core::marker::PhantomData; struct Pin; impl Pin { fn set_high(&self) { // Set pin N high } } // No runtime overhead - pin number is compile-time constant let led = Pin::<5>; // LED on pin 5 led.set_high();

10. Bit Manipulation Syntax

Common patterns for register manipulation:

// Set bit register |= 1 << 5; // Clear bit register &= !(1 << 5); // Toggle bit register ^= 1 << 5; // Check bit if register & (1 << 5) != 0 { // Bit is set } // Set multiple bits register |= (0b1010 << 4); // Clear multiple bits and set new value register = (register & !0xFF) | 0x42; // Using bitfield syntax (more readable) register .modify(|_, w| w .bit5().set_bit() .bit3().clear_bit() );

11. Match for Register Decoding

let status = read_status_register(); match status { 0x00 => { /* Ready */ }, 0x01 => { /* Busy */ }, 0x02 => { /* Error */ }, _ => { /* Unknown */ }, } // With bit patterns match status & 0b11110000 { 0b0000_0000 => { /* Mode 0 */ }, 0b0001_0000 => { /* Mode 1 */ }, 0b0010_0000 => { /* Mode 2 */ }, _ => { /* Invalid */ }, }

12. Associated Constants

trait MemoryLayout { const BASE_ADDRESS: usize; const SIZE: usize; } struct Flash; impl MemoryLayout for Flash { const BASE_ADDRESS: usize = 0x0800_0000; const SIZE: usize = 256 * 1024; // 256 KB } // Use at compile time const FLASH_END: usize = Flash::BASE_ADDRESS + Flash::SIZE;

13. Enums for State Machines

enum UartState { Idle, Transmitting { byte: u8, bit_index: u8 }, Receiving { byte: u8, bit_index: u8 }, Error(UartError), } enum UartError { FramingError, ParityError, Overrun, } impl UartState { fn process(&mut self) { match self { UartState::Idle => { /* Wait for start bit */ }, UartState::Transmitting { byte, bit_index } => { // Transmit next bit }, UartState::Receiving { byte, bit_index } => { // Receive next bit }, UartState::Error(e) => { // Handle error }, } } }

14. Useful Macros for Embedded

concat! for String Building

const VERSION: &str = concat!("v", env!("CARGO_PKG_VERSION"));

include_bytes! for Embedding Data

const LOGO: &[u8] = include_bytes!("logo.bin");

cfg! for Conditional Compilation

#[cfg(feature = "stm32f303")] use stm32f3xx_hal as hal; #[cfg(feature = "stm32f401")] use stm32f4xx_hal as hal; const CLOCK_SPEED: u32 = if cfg!(feature = "stm32f303") { 72_000_000 } else { 84_000_000 };

15. Embedded-Friendly Patterns

Option for Nullable Hardware References

struct System { uart: Option, spi: Option, } impl System { fn send_uart(&mut self, data: &[u8]) -> Result<(), Error> { if let Some(uart) = &mut self.uart { uart.write(data) } else { Err(Error::NotInitialized) } } }

Result for Error Handling

enum I2cError { Busy, ArbitrationLoss, Nack, Timeout, } fn i2c_write(addr: u8, data: &[u8]) -> Result<(), I2cError> { if is_busy() { return Err(I2cError::Busy); } // Perform write Ok(()) }

Best Practices

  • Minimize unsafe: Use HAL abstractions when possible
  • Document safety: Always explain unsafe code with comments
  • Use type system: Leverage compile-time checks (const, generics, type states)
  • Prefer safe wrappers: Create safe APIs over unsafe primitives
  • Volatile for MMIO: Always use volatile operations for hardware registers
  • Match for decoding: Use pattern matching for register interpretation
  • Zero-cost abstractions: Leverage ZSTs and const generics
