Configuring Toolchains & Environment

After installing the Rust toolchain and embedded targets, proper configuration is essential for efficient embedded development. This guide covers toolchain management, cross-compilation setup, linker configuration, and environment optimization.

Understanding Toolchains

A Rust toolchain consists of several components working together:

  • rustc: The Rust compiler that generates machine code
  • cargo: Build system and package manager
  • std/core: Standard or core library for your target
  • rustfmt: Code formatter for consistent style
  • clippy: Linter for catching common mistakes

Toolchain Channels

Rust offers three release channels, each serving different purposes:

Channel Update Frequency Use Case Stability
Stable Every 6 weeks Production code, recommended for most projects Highest - fully tested releases
Beta Same as stable Testing upcoming features before stable release High - release candidates
Nightly Daily Cutting-edge features, experimental APIs Variable - may have bugs

Installing Different Channels

rustup install stable rustup install beta rustup install nightly

Setting Default Toolchain

rustup default stable rustup default nightly # If you need nightly features

Per-Project Toolchain Override

Use a specific toolchain for a particular project:

cd my-embedded-project rustup override set stable rustup override set nightly # For projects needing nightly
Recommendation: Use stable channel for embedded projects unless you specifically need nightly features. The embedded ecosystem is well-supported on stable Rust.

Cross-Compilation Configuration

Understanding Cross-Compilation

Cross-compilation means building code on one architecture (your computer) to run on a different architecture (microcontroller). This requires proper toolchain configuration.

Project Configuration File: .cargo/config.toml

Create a .cargo/config.toml file in your project root to configure build behavior:

Example Configuration for STM32F3 (Cortex-M4F)

[build] # Set default compilation target target = "thumbv7em-none-eabihf" [target.thumbv7em-none-eabihf] # Use custom linker script rustflags = [ "-C", "link-arg=-Tlink.x", "-C", "link-arg=--nmagic", ] # Use rust-lld linker (LLVM linker shipped with Rust) linker = "rust-lld" [target.'cfg(all(target_arch = "arm", target_os = "none"))'] # Runner for executing code on hardware/emulator runner = "probe-rs run --chip STM32F303VCTx"

Configuration for QEMU Emulation

[build] target = "thumbv7m-none-eabi" [target.thumbv7m-none-eabi] runner = "qemu-system-arm -cpu cortex-m3 -machine lm3s6965evb -nographic -semihosting-config enable=on,target=native -kernel"

Linker Script Configuration

Embedded systems require custom memory layouts defined in linker scripts. The cortex-m-rt crate provides a linker script template.

# In Cargo.toml [dependencies] cortex-m = "0.7" cortex-m-rt = "0.7"

Create memory.x in your project root defining memory regions:

MEMORY { /* STM32F303VC memory layout */ FLASH : ORIGIN = 0x08000000, LENGTH = 256K RAM : ORIGIN = 0x20000000, LENGTH = 40K }
Where to find memory layout info: Consult your microcontroller's datasheet or reference manual. Key sections: "Memory organization" or "Memory mapping".

Environment Variables

Important Rust Environment Variables

Variable Purpose Example
RUSTFLAGS Pass flags to rustc compiler -C link-arg=-Tlink.x
CARGO_BUILD_TARGET Set default build target thumbv7em-none-eabihf
DEFMT_LOG Set logging level for defmt debug, info, warn
PROBE_RS_CHIP Specify target chip for probe-rs STM32F303VCTx

Setting Environment Variables

Linux/macOS (temporary):

export RUSTFLAGS="-C link-arg=-Tlink.x" export CARGO_BUILD_TARGET=thumbv7em-none-eabihf

Linux/macOS (permanent - add to ~/.bashrc or ~/.zshrc):

echo 'export CARGO_BUILD_TARGET=thumbv7em-none-eabihf' >> ~/.bashrc source ~/.bashrc

Windows PowerShell:

$env:RUSTFLAGS="-C link-arg=-Tlink.x" $env:CARGO_BUILD_TARGET="thumbv7em-none-eabihf"

Managing Multiple Targets

Listing Installed Targets

rustup target list --installed

Building for Specific Target

cargo build --target thumbv7em-none-eabihf cargo build --target thumbv6m-none-eabi cargo build --target riscv32imac-unknown-none-elf

Target-Specific Dependencies

Configure dependencies for specific targets in Cargo.toml:

[dependencies] cortex-m = "0.7" # Only include on ARM targets [target.'cfg(target_arch = "arm")'.dependencies] cortex-m-rt = "0.7" # Only include on RISC-V targets [target.'cfg(target_arch = "riscv32")'.dependencies] riscv-rt = "0.11"

Workspace Configuration

Setting Up a Cargo Workspace

For projects with multiple embedded targets or shared code:

# Workspace Cargo.toml at root [workspace] members = [ "firmware-stm32", "firmware-nrf", "shared-drivers", ] [workspace.dependencies] # Shared dependencies embedded-hal = "1.0" defmt = "0.3"

Using Workspace Dependencies

# In member Cargo.toml [dependencies] embedded-hal = { workspace = true } defmt = { workspace = true }

Optimization Settings

Build Profiles

Configure optimization levels in Cargo.toml:

[profile.dev] # Fast compilation, larger binary, no optimizations opt-level = 0 debug = true [profile.release] # Optimized for speed, smaller binary opt-level = 3 debug = false lto = true # Link-Time Optimization [profile.release-with-debug] # Optimized but with debug symbols inherits = "release" debug = true [profile.release-size] # Optimized for binary size inherits = "release" opt-level = "z" # Optimize for size lto = true codegen-units = 1 # Better optimization, slower build strip = true # Remove debug symbols
Optimization Levels:
  • 0 = No optimization (fastest compile, largest binary)
  • 1 = Basic optimization
  • 2 = Standard optimization
  • 3 = Aggressive optimization (slowest compile, best performance)
  • s = Optimize for size
  • z = Aggressively optimize for size

Feature Flags and Conditional Compilation

Defining Features

# Cargo.toml [features] default = ["stm32f303"] stm32f303 = [] stm32f401 = [] use-semihosting = []

Using Features in Code

#[cfg(feature = "stm32f303")] use stm32f3xx_hal as hal; #[cfg(feature = "stm32f401")] use stm32f4xx_hal as hal; #[cfg(feature = "use-semihosting")] use cortex_m_semihosting::hprintln;

Building with Features

cargo build --features stm32f303 cargo build --features "stm32f401,use-semihosting" cargo build --no-default-features --features stm32f401

Toolchain Components

Adding Components

# Add rustfmt for code formatting rustup component add rustfmt # Add clippy for linting rustup component add clippy # Add rust-src for building core library rustup component add rust-src # Add LLVM tools rustup component add llvm-tools

Using Components

# Format code cargo fmt # Run linter cargo clippy # Check binary size cargo size --release -- -A # Disassemble binary cargo objdump --release -- -d > output.asm

Troubleshooting Common Configuration Issues

Issue: "error: linking with `rust-lld` failed"

Possible causes:
  • Missing memory.x linker script
  • Incorrect memory layout in memory.x
  • Missing cortex-m-rt dependency
Solution: Verify memory.x exists and has correct memory regions. Ensure cortex-m-rt is in dependencies.

Issue: "cannot find -lc"

Cause: Trying to link against standard C library in no_std environment.
Solution: Use #![no_std] and avoid dependencies that require std.

Issue: Target not found

Solution: Install the target:
rustup target add thumbv7em-none-eabihf

Best Practices

  • Keep .cargo/config.toml in version control for consistent team builds
  • Document required toolchain version in README
  • Use workspace dependencies for multi-target projects
  • Enable LTO for production releases to reduce binary size
  • Use feature flags to support multiple hardware variants
  • Keep toolchain updated: rustup update
  • Use stable channel unless nightly features are essential
